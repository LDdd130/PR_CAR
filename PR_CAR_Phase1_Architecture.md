## 0. 현황 진단 — 저장소 분석과 Phase 1 갭

### 0.1 코드베이스 구조 (HEAD 기준)

`main.c`는 초기화 + 커널 기동만 담당하고 애플리케이션 로직은 전부 모듈로 분리돼 있다. 계층이 이미 잘 잡혀 있으므로 Phase 1은 "구조 재편"이 아니라 **기존 계층 위에 얹는 작업**으로 진행한다.

| 계층 | 파일 | 책임 |
|---|---|---|
| 부팅/초기화 | `main.c` | HAL·페리프 init + 스케줄러 기동만 |
| 태스크/IPC | `freertos.c` | 3태스크 정의, `driveQ`/`uartQ`, ToF 듀얼 기동(`Init_ToF_Sensors`) |
| 주행 상태머신 | `drive.c/.h` | CRUISE·BRAKE·SPIN·REVERSE·HOLD·SIDE_AVOID·CORNER 7상태, 45° 그리드 스냅 heading, 튜닝 노브 단일 블록 |
| 센서 드라이버 | `ultra.c` (전방 IC), `vl53l0x.c` (경량 ToF), `bno055.c` |
| 액추에이터 | `motor.c/.h` | 속도 % 0~100 → TIM4 CCR 0~999, `Car_*` 프리미티브 |
| 안전 | IWDG 2.048 s | MotorTask가 '신선한 큐 수신 성공' 시에만 refresh |

리소스 점유 현황:

| 자원 | 용도 | 설정 |
|---|---|---|
| TIM3 | 전방 에코 입력캡처 CH1 (PA6) | 1 µs tick, 16-bit, IRQ prio 5 |
| TIM4 | 모터 PWM CH1/CH2 (PB6/PB7) | 1 kHz, ARR 999 |
| TIM11 | `delay_us` 베이스 | 1 µs free-run |
| TIM1 · **TIM2(32-bit)** · **TIM5(32-bit)** · TIM9 | 미사용 | 확장 여유 |
| I2C1 (PB8/PB9) | ToF L(0x60) · ToF R(0x52) · BNO055 | 100 kHz Standard |
| USART1 (PA9/PA10) | HC-06 블루투스 — 현재 **RX 전용 운용** | 9600-8N1, 1바이트 IT 수신 |
| FreeRTOS | heap 15,360 B / 태스크 스택 각 256 word | MotorTask=Normal, Sensor·BT=Low |

### 0.2 Phase 1 갭 분석 — 가장 중요한 발견

| Phase 1 목표 | 저장소 실상 (HEAD 기준) | 본 설계서의 취급 |
|---|---|---|
| ① 측면 초음파 제거 + ToF×2 (XSHUT 주소 분리) | **이미 구현 완료.** PA1/PA2 XSHUT, 좌 0x60·우 0x52, 타이밍버짓 20 ms 연속측정, 센터링 PD(`Drive_CenteringPD_Run`)까지 가동 중 | as-built 검증 + 설계 근거의 공식 문서화 (§1) |
| ② 휠 엔코더(Input Capture) + 속도 PID | **미구현.** 현재 개루프 %duty 제어 | 신규 설계 (§1.3, §2, §3.2) |

**진단 코멘트.** 현 시스템의 최대 기술부채는 "duty를 명령하지만 실제 속도는 아무도 모른다"는 개루프 구조다. 배터리 전압·바닥 마찰에 따라 같은 duty가 다른 속도를 내며, 이것이 HANDOVER.md에 기록된 **긴 직선 지그재그(리밋사이클)와 45° 코너 과회전의 공통 뿌리** 중 하나다 — 조향 게인이 실속도에 따라 과대/과소 작동하기 때문이다. 따라서 Phase 1의 무게중심은 ②이고, ③은 ②의 튜닝 계측 도구로서 **선행 배치**를 권한다(§5 로드맵).

## 1. 하드웨어 리소스 및 핀맵 정리 가이드

### 1.1 초음파→ToF 전환으로 회수된 자원 (커밋 `677a1ea` → `33e5b24` 실측 diff)

| 자원 | 구(측면 초음파 시절) | 현재 | 상태 |
|---|---|---|---|
| PA3 | `TRIG_RIGHT` (GPIO 출력) | 미사용 | **회수** |
| PA4 | `TRIG_LEFT` (GPIO 출력) | 미사용 | **회수** |
| PA7 | `ECHO_TIM3_CH2` (좌 에코 캡처) | 미사용 | **회수** |
| PB0 | `ECHO_TIM3_CH3` (우 에코 캡처) | 미사용 | **회수** |
| TIM3_CH2 / CH3 | 측면 에코 입력캡처 채널 | 유휴 | **회수** |
| PA1 / PA2 | 미사용 | `TOF_LEFT_XSHUT` / `TOF_RIGHT_XSHUT` | 신규 점유 |

### 1.2 이 회수가 엔코더(Timer Input Capture) 확장에 갖는 아키텍처적 의미

**(1) IC 가능 채널의 희소성 해소.** 48핀 패키지에서 타이머 채널은 핀과 1:1로 묶여 있고, I2C·UART·SWD·모터 IN이 선점한 뒤 남는 입력캡처 가능 핀은 거의 없다. 회수된 TIM3_CH2/CH3은 "이미 입력캡처로 검증된 타이머의, 이미 하네스가 지나가던 핀"이라는 점에서 최상급 자원이다.

**(2) 시간베이스 재사용 — PWM 타이머와 분리된 IC 전용 타이머 확보가 핵심.** 에코 펄스폭 측정과 엔코더 주기 측정은 요구 해상도가 같다(µs). TIM3은 이미 1 µs tick으로 돌고 있으므로 채널만 켜면 된다. 반대로 이를 PWM 타이머(TIM4)에 얹으려 하면 ARR이 PWM 주파수(1 kHz)에 묶여 캡처 레인지가 1 ms로 강제된다. 초음파를 걷어내며 얻은 실질 가치는 핀 몇 개가 아니라 **"프리스케일러를 측정 목적에 전용(專用)할 수 있는 타이머"** 그 자체다.

**(3) ISR 인프라 재사용.** `HAL_TIM_IC_CaptureCallback`은 이미 instance/channel로 분기하는 구조이고, `ultra.c` 주석에 "추후 타이머 추가 대비"라고 명시돼 있다. 엔코더 좌/우는 분기 2개 추가로 흡수된다 — 새 ISR 뼈대가 필요 없다.

**(4) 인터럽트·시간구조 예산의 재배치.** 측면 초음파는 측정 1회당 TRIG 스케줄 + 에지 인터럽트 2회에 더해, 크로스토크 회피를 위한 **순차발사 강제(front→left→front→right)**라는 시간구조 비용을 SensorTask에 부과했다. ToF는 측정 자체를 센서 코어로 오프로드했고, 이렇게 비운 예산이 엔코더 에지 IRQ(주행 시 바퀴당 약 20~100 Hz, §3.2)를 여유 있게 흡수한다. 순차발사 제약이 사라져 센서 루프 리듬이 결정적으로 변한 것도 중요하다 — 이는 속도 PID가 원하는 "일정한 데이터 흐름"의 전제조건이다.

**(5) 배선 연속성.** 구 에코 배선 경로(PA7/PB0)를 엔코더 신호선이 그대로 승계하므로 하네스 변경이 최소화된다.

### 1.3 엔코더 채널 배치 — 2안 비교와 권장

| 기준 | **A안: TIM3_CH2(PA7) + TIM3_CH3(PB0)** | B안: TIM2_CH3(PB10) + TIM2_CH4(PA3), 32-bit |
|---|---|---|
| .ioc 변경량 | 기존 TIM3에 IC 채널 2개 추가만 | 타이머 신규 init + IRQ 신설 |
| 카운터 폭 | 16-bit @1 µs → 65.5 ms에서 랩 | 32-bit → 71분 무랩 |
| 저속 한계 | Update IT 기반 16→32-bit 소프트 확장 필요. 미적용 시 약 0.16 m/s 미만 모호 | 산술 단순 |
| ISR | 기존 TIM3 IRQ 1개에 통합 | TIM2 IRQ 추가 |
| 배선 | 회수 핀 직승계 | PB10 신규 배선 |

**권장: A안.** 근거 — 0속 판정 타임아웃(§3.2)은 어느 안이든 필수라서 B안의 "단순함" 이득은 보기보다 작고, A안은 타이머·ISR·배선 재사용을 극대화한다. 단, Update IT 오버플로 확장을 R1 완료 기준에 반드시 포함한다. 팀이 랩 산술이 부담스러우면 B안으로 바꿔도 상위 설계(§2~§4)는 전혀 흔들리지 않는다 — 측정 모듈 인터페이스 뒤에 숨기면 된다.

### 1.5 VL53L0X 듀얼 주소 할당 — XSHUT 제어 논리 시퀀스

전제 3가지: (a) 모든 VL53L0X는 부팅 시 7-bit 0x29(=8-bit 0x52) 고정 주소로 깨어난다. (b) 새 주소는 레지스터 0x8A에 기록하는 **내부 RAM 값 — 휘발성**이다. (c) XSHUT=Low는 HW standby로서 주소가 소실되고 측정이 멈추며, High 복귀 후 t_boot ≈ 1.2 ms의 부팅 시간이 필요하다.

| 단계 | 조작 | 그 시점의 I2C 버스 | 목적 |
|---|---|---|---|
| S0 | XSHUT_L=Low, XSHUT_R=Low 동시, ≥10 ms 유지 | ToF 없음 | **임의의 선행 상태를 무효화하는 '소프트웨어 파워사이클'** |
| S1 | XSHUT_L=High, ≥10 ms (t_boot 여유) | 좌측만 0x52 | 좌측 단독 노출 |
| S2 | 0x52 대상으로 reg 0x8A ← 0x30(7-bit) 기록 | 좌측이 0x60(8-bit)으로 이동 | **충돌이 발생하기 전에 먼저 주소를 옮긴다** |
| S3 | 좌측 Init (SPAD·기준 캘리브레이션) | — | 새 주소로 통신 검증 겸 초기화 |
| S4 | XSHUT_R=High, ≥10 ms | 좌 0x60 + 우 0x52 | 이제 0x52는 우측 유일 → 충돌 없음 |
| S5 | 우측 Init (디폴트 0x52 유지) | — | 재배치 1회로 충분 — 한쪽만 옮기면 된다 |
| S6 | 양측 타이밍버짓 20 ms + 연속측정 시작 | — | 제어루프(~20 ms)와 샘플레이트 정합 |

**as-built 대조:** 현 `Init_ToF_Sensors()`는 위 S0~S6과 정확히 일치하며, 주소 기록이 성공했을 때만 핸들의 주소를 갱신(실패 시 구 주소로 통신 지속)하는 방어까지 갖춰져 있다. **검증 통과.**

불변식 4가지:

1. **운용 중 XSHUT는 항상 High.** 순간이라도 Low가 되면 주소 소실 + 측정 정지.
2. **매 부팅 재수행 필수 — 특히 IWDG 리셋 경로가 진짜 이유다.** 워치독 리셋은 MCU만 리셋하고 센서 전원은 유지된다. 즉 리셋 직후 좌측 센서는 0x60을 "기억한 채" 남아 있는 비대칭 상태다. S0의 동시 Low가 어떤 선행 상태든 항상 초기상태로 수렴시킨다. 이 프로젝트처럼 IWDG를 상시 무장한 시스템에서 XSHUT 제어는 주소 분리 수단을 넘어 **결정적(deterministic) 리셋 경로**로서 필수다.
3. **전기 사양:** VL53L0X 코어의 XSHUT는 2.8 V 도메인. 레벨시프터 내장 브레이크아웃(GY-530류)이면 3.3 V push-pull 직결 가능, 미내장 순정 모듈이면 open-drain + 풀업 구동. → §6 체크리스트.
4. **주소 지형 경고:** 우측 ToF의 0x52(8-bit)는 **BNO055의 대체 주소(7-bit 0x29)와 같은 값**이다. 현 구성이 동작한다는 사실 자체가 BNO055가 0x28(=0x50)에 있다는 증거다. BNO055의 ADR 스트랩을 절대 바꾸지 말 것 — 금지 항목으로 명문화한다.

## 2. FreeRTOS Task 아키텍처 및 데이터 흐름 설계

### 2.1 계승할 설계 불변식 (현 코드의 자산)

Phase 1에서 무엇을 추가하든 아래 3원칙은 그대로 유지한다. 현 코드가 확립했고 실전(워치독 리셋, 큐 단절)에서 검증된 원칙이다.

1. **모터 명령 발행자는 MotorTask 유일** (폴트 핸들러의 비상정지 제외) — 경쟁의 원천 차단.
2. **IWDG refresh는 '신선한 센서 데이터가 흐를 때만'** — SensorTask 사망, 큐 단절, MotorTask 행(hang) 어느 쪽이든 2.048 s 내 리셋.
3. **뮤텍스 0개 유지** — 다필드 스냅샷은 값복사 큐로, 원자폭 이하 상태는 단일작성자 volatile로. Phase 1 신규 데이터도 전부 이 2계층 안에서 해결한다(§2.4).

### 2.2 태스크 R&R 정의

| 태스크 | 우선순위 | 트리거/주기 | 해야 하는 일 (이것이 전부) | 해서는 안 되는 일 |
|---|---|---|---|---|
| **SensorTask** | Low | 자체 루프 ~20 ms (ToF 버짓 정합) | 전방 초음파 발사·에코 대기, ToF L/R 논블로킹 폴링, BNO055 heading 폴링, median 필터·유효성 게이팅(miss/stale), `DriveInputs` 스냅샷 발행(early 선제 메시지 포함), 텔레메트리 **센서 필드** 미러 갱신 | 제어 판단(상태머신), 모터 호출, UART 송신, 무거운 부동소수 연산 |
| **MotorTask** | Normal (최고) | `driveQ` 수신 + timeout 10 ms 하이브리드 (§2.3) | 매 회: 엔코더 스냅샷 → 속도 계산(T법) → **속도 PI → PWM 인가**. din 신선 시: `Drive_Update`로 목표속도 갱신. din 나이 150 ms 초과 시 Car_Stop + refresh 생략(기존 fail-safe 의미 보존). IWDG refresh. 텔레메트리 **모터 필드** 미러 갱신 | I2C·센서 직접 접근, UART, 10 ms를 넘는 블로킹 |
| **BluetoothTask** | Low | `uartQ` 수신(가변 timeout) + 100 ms TX 스케줄 | RX 파싱(레거시 1바이트 + `#` 라인), 튜닝 파라미터 원자 반영, 텔레메트리 프레임 조립·IT 송신 | 모터 직접 호출(기존대로 `sys_*`/`manual_*` 변수 경유만), 센서 접근 |
| **ISR 계층** | HW | 하드웨어 이벤트 | TIM3_CC1 에코 에지, **TIM3_CC2/CC3 엔코더 에지(기록만)**, **TIM3_UP 오버플로 카운트**, USART1 RXNE 1바이트→uartQ, TX 완료 플래그 | 계산·필터·판단 일체 — 기록 후 즉시 복귀 |

조정 권고 2가지: (a) SensorTask를 BluetoothTask보다 한 단계 위(예: Low1)로 분리해 텔레메트리 송신이 측정 사이클을 밀지 못하게 한다. (b) BluetoothTask 스택을 256→384 word로 상향(프레임 문자열 버퍼) 후 high-water 재측정. **float printf 금지** — heading 등은 ×10 정수 표기로 회피(§4.1).

### 2.3 MotorTask 루프 구조 — 이벤트 구동에서 하이브리드로

속도 PI는 고정 주기가 이상적이지만, 현 아키텍처가 투자한 "early 선제제동 메시지 → 고우선 MotorTask 즉시 선점" 자산을 버려선 안 된다. 절충 설계:

- `driveQ` 수신 timeout을 기존 150 ms에서 **10 ms**로 내린다 → din이 오면 즉시 깨고(제동 반응 지연 0 유지), 없어도 최대 10 ms마다 깨어 속도루프를 돈다.
- PI의 dt는 고정 상수가 아니라 **틱 실측값**(직전 실행과의 차)을 사용한다. 5~15 ms 지터는 속도루프 대역폭에서 무해하다.
- fail-safe는 "마지막 din 수신 시각"을 별도 추적해 150 ms 초과 시 Car_Stop + refresh 생략 — 기존 IWDG 의미를 1:1 보존한다.

### 2.4 데이터 흐름(Data Flow) 설계

```
[ISR 계층]
 TIM3_CC1 (에코 µs) ─┐              TIM3_CC2/CC3 (엔코더 에지) + TIM3_UP (ovf)
                     │                              │ D2: volatile 기록
                     ▼                              ▼
 I2C1 ───▶ ┌─────────────┐   D1: driveQ    ┌───────────────────┐
 ToF L/R   │ SensorTask  │  메일박스(값복사) │     MotorTask     │──▶ TIM4 PWM
 BNO055    │ 측정·필터·  │ ──────────────▶ │ ① 속도 계산(T법)   │    + IN1~4
           │ 게이팅      │  DriveInputs    │ ② 속도 PI → PWM   │
           └──────┬──────┘                 │ ③ Drive_Update    │
                  │ D4: 센서 미러(volatile) │ ④ IWDG(신선 게이트)│
                  ▼                        └─────────┬─────────┘
            [텔레메트리 미러] ◀── D3: 모터 미러 ───────┘
                  │                                  ▲
                  │ 10 Hz 프레임 조립                 │ D6: sys_*/manual_*/튜닝값
                  ▼                                  │     (volatile, BT가 작성)
 USART1 RX ─▶ uartQ(FIFO 16) ─▶ ┌───────────────┐ ──┘
   (D5: 1바이트 IT)              │ BluetoothTask │──▶ USART1 TX(IT) ─▶ HC-06 ─▶ 앱
                                └───────────────┘
```

채널별 IPC 메커니즘 선택 근거:

| # | 경로 | 데이터 | 메커니즘 | 선택 근거 |
|---|---|---|---|---|
| D1 | Sensor → Motor | `DriveInputs` (f, l, r, valid류, heading, now) | **driveQ 값복사 메일박스 depth 2** (기존 유지) | 다필드가 '같은 시점' 스냅샷이어야 하는 유일한 경로 → 큐 복사가 정답. depth 2 = full + early 선제 메시지 공존 |
| D2 | 엔코더 ISR → Motor | 바퀴당 최근 캡처값·ovf·에지 seq | **volatile + 짧은 임계구역 스냅샷** | ISR은 기록만(최단화), 계산은 태스크가. 캡처값·ovf·seq 3워드의 정합만 critical section으로 보호 |
| D3 | Motor → BT | vL, vR, state, pwm | **단일작성자 volatile 미러** | 16/32-bit 정렬 store는 Cortex-M4에서 원자. 표시용이라 필드 간 ≤1사이클 시차 허용 → 락프리 |
| D4 | Sensor → BT | front, dL, dR (기존 `dist_left/right` 미러 확장), hdg_x10, flags | 상동 | 상동 |
| D5 | UART RX ISR → BT | 명령 바이트 | **uartQ FIFO 16** (기존) | 명령은 이벤트 — 유실 불가 → 최신값 덮어쓰기가 아니라 FIFO가 맞다 |
| D6 | BT → Motor | sys_power/mode, manual_L/R (기존) + v_target·게인 (신규) | **단일작성자 volatile** | 8/16/32-bit 정렬 단일 store = 원자. float 게인도 32-bit 정렬이면 동일 |

**판별식 한 줄 요약: "제어 판단에 함께 들어가는 다필드 묶음 = 큐 스냅샷(D1) / 원자폭 이하의 상태 값 = 단일작성자 volatile(D2~D4, D6) / 유실 불가 이벤트 = FIFO(D5)."** 이 세 문장이 본 시스템 IPC의 전부이며, 뮤텍스는 계속 0개다.

신규 데이터 모델 (필드 수준 정의 — 구현은 마일스톤에서):

| 구조/영역 | 소유자(writer) | 필드 | 비고 |
|---|---|---|---|
| WheelCapture[2] | 엔코더 ISR | last_cap(16b), ovf_snapshot, edge_seq | seq 증가 여부로 '새 에지' 판별 |
| SpeedState[2] | MotorTask | period_us(32b), v_meas, v_filt [cm/s] | 0속 타임아웃 상태 포함 |
| SpeedPI[2] | MotorTask | v_target, 적분항, duty_out | anti-windup 상태 포함 |
| TelemetryMirror | Sensor·Motor 분할 | §4.2 프레임 필드와 1:1 | 필드마다 단일작성자 원칙 준수 |
| TuneParams | BluetoothTask | v_target_cmps, kp_e, kh, kd_yaw, tel_on, tel_hz | Motor/BT가 읽기 전용 |

`DriveInputs`는 확장하지 않는다 — 센서 도메인의 순수성을 유지한다. 속도는 MotorTask가 자체 생산·소비하므로 큐를 오갈 이유가 없다.

## 3. 알고리즘 개발 방향성

### 3.1 벽 타기(중앙 유지) 제어 논리 — 핵심 원리

**기하부터 시작한다.** 복도 폭을 W, 차체 중심의 횡편차를 y(좌측 치우침 +)라 하면, 측면 센서가 차체 대칭 장착일 때:

> dL ≈ W/2 − y,  dR ≈ W/2 + y  →  **e = dL − dR = −2y**

좌우 ToF의 차이는 그 자체로 "중앙에서 벗어난 정도"의 2배 척도다. 현 코드의 `wall_error = dist_left − dist_right`와 동일한 정의이며(e > 0 = 좌측이 트임 = 차가 우벽 쪽), **W를 몰라도 된다**는 것이 이 방식의 힘이다 — 코스 폭이 37~67 cm로 섞여 있어도 목표는 언제나 e → 0.

**e의 시간미분이 곧 heading이다.** 코스축 대비 진행각 오차를 ψe라 하면 ẏ = v·sin ψe이므로:

> ė = −2v·sin ψe ≈ −2v·ψe (소각 근사)

이 등식이 설계의 핵심 통찰이다. **거리오차의 D항을 ToF를 미분해서(잡음 증폭) 만들 필요가 없다 — BNO055가 같은 정보를 훨씬 깨끗한 신호로 준다.** 현 코드의 KD_YAW(요레이트 감쇠)·KP_HDG(그리드축 정렬)가 이미 이 역할을 수행 중이며, Phase 1은 이 구조를 유지·강화한다.

**제어식(3항 구성)과 부호 규약.** 조향량 u (+ = 좌회전):

> u = Kp_e·(dL − dR) + Kh·(ψ_grid − ψ) − Kd·ψ̇

| 항 | 물리 역할 | 현 코드 대응 |
|---|---|---|
| Kp_e·e | 횡위치 복원 (중앙으로 끌어당김) | 센터링 PD의 P |
| Kh·(ψ_grid − ψ) | 코스축 정렬 (45° 그리드 스냅 기준각) | KP_HDG = 0.58 |
| −Kd·ψ̇ | 감쇠 (지그재그 리밋사이클 억제) | KD_YAW = 0.30 |

**차동 배분:** vt_L = v_base − u, vt_R = v_base + u. u > 0이면 우바퀴가 빨라져 좌회전. 여기서 결정적 전환이 일어난다 — 현재 u의 단위는 %duty지만, 속도루프 도입 후에는 **cm/s라는 물리량**이 된다.

**속도루프가 지그재그를 어떻게 개선하는가.** IMG_2986의 직선 지그재그는 전형적인 리밋사이클로, (측정 ~20 ms + ToF 버짓 + 구동 지연) × 높은 게인 × **duty 비선형(정지마찰 데드밴드, 배터리 전압 의존)**의 합작이다. u가 Δduty인 동안에는 같은 u라도 만충/저충, 저속/고속에서 실제 요레이트가 달라져 게인을 한 점에 고정할 수 없다. u가 Δ속도[cm/s]가 되고 내부 PI가 그 속도를 실제로 만들어 주면 **조향 게인이 배터리·속도와 무관해져** 한 번 맞춘 튜닝이 유지된다. Kp_e를 낮추고 Kh/Kd 비중을 키우는 재튜닝(R5)과 병행한다.

**예외 계층 (기존 로직과의 접속점):**

| 상황 | 판정 | 대응 |
|---|---|---|
| 한쪽 벽 소실 (개구부·코너 진입) | 해당 측 > SIDE_OPEN 또는 캡(1000 mm) 도달 | 위치항 무효화 → 유효벽 기준 목표거리 유지: e′ = ±(d_ref − d_wall). 양쪽 모두 소실이면 순수 heading-hold (기존 '트임 만료' 로직 그대로) |
| 단발 스파이크 (유리·저반사면) | — | 기존 median(SIDE_MED_WIN) 필터 계층 유지 |
| 새 샘플 정체 | TOF_STALE 150 ms 초과 | side_valid = 0 강등 (기존 게이팅 그대로) |
| 요각 시 측정 과대 | 측정거리 = 실거리 / cos ψe | ψe < 10°면 오차 < 1.5% → Phase 1에서는 생략 가능. d·cos ψe 보정은 Phase 2 옵션 |
| ToF FOV 25° 블렌딩 | 15 cm 거리에서 스팟 지름 ~6.6 cm — 코너 모서리에서 좌우 값이 섞임 | 코너 근방 측면값 신뢰 강등은 기존 상태머신(CORNER 상태)이 담당 — 벽타기는 CRUISE 한정 |

**전체 캐스케이드 구조:**

```
[외루프: 조향]  센서 이벤트레이트 ~20 ms       [내루프: 속도]  10 ms
 e, ψ ──▶ u [cm/s] ──▶ vt_L = v_base − u ──▶ 좌 휠 PI ──▶ duty_L ──▶ PWM
                       vt_R = v_base + u ──▶ 우 휠 PI ──▶ duty_R ──▶ PWM
                                                ▲ 엔코더 v_meas (T법 측정)
```

내루프가 외루프보다 2배 이상 빨라야 캐스케이드가 안정된다 — 10 ms vs ~20 ms로 충족한다.

### 3.2 엔코더 기반 속도 제어 — 제어 흐름 기획

**가정 (착수 전 실측으로 치환, §6.2):** 휠 Ø65 mm (원주 C ≈ 20.4 cm), 슬롯 N = 20 → 펄스당 이동 ≈ 1.02 cm.

**왜 Input Capture(T법)인가 — 정량 근거.** 속도 측정은 M법(고정 창에서 펄스 수 세기)과 T법(펄스 간 시간 측정)이 있다.

| 차속 | 펄스 주파수 | M법: 10 ms 창당 펄스 수 | T법: 주기 (1 µs 분해능) |
|---|---|---|---|
| 0.2 m/s | 19.6 Hz | 0.196개 (±100% 양자화) | 51.0 ms |
| 0.5 m/s | 49 Hz | 0.49개 | 20.4 ms |
| 1.0 m/s | 98 Hz | 0.98개 | 10.2 ms |

저PPR(20슬롯)에서는 제어주기당 펄스가 1개도 안 돼 M법의 양자화 잡음이 신호를 삼킨다. T법은 같은 조건에서 1/10⁴ 수준 분해능을 준다. **Input Capture 선택은 정당하며, 이 하드웨어 조건에서는 사실상 유일해다.**

**측정 파이프라인 (설계 순서):**

1. **캡처 (ISR):** 상승 단일 에지만 사용 — 슬롯 듀티가 비대칭이라 양 에지를 쓰면 주기가 교대로 출렁인다. ICFilter를 0x3~0xF로 올려 글리치를 하드웨어단에서 억제.
2. **16→32-bit 확장 (ISR):** TIM3 Update IT에서 오버플로 카운트 증가. 주기 = (에지 간 ovf 차) × 65536 + 캡처값 차. 이것으로 저속 한계(≈0.16 m/s)가 사라진다.
3. **주기 → 속도 환산 (MotorTask):** v[cm/s] = (C/N) × 10⁶ / T[µs] ≈ 1.02×10⁶ / T. (검산: T = 10,200 µs → 100 cm/s)
4. **0속 판정:** 마지막 에지 후 100 ms 무이벤트 → v = 0 강제. "멈추면 다음 에지가 영원히 안 온다"는 T법의 구조적 맹점을 막는 필수 보완이다.
5. **필터:** 주기값에 median3 또는 EMA(α ≈ 0.3). 주 잡음원은 슬롯 가공 공차.
6. **방향:** 단채널 포토센서는 방향 정보가 없다 → 속도 부호는 '명령 방향'을 채택하고 이 한계를 명문화한다 (쿼드러처 아님).

**속도 제어기 — PI + 피드포워드:**

> duty = FF(v_target) + Kp·(v_target − v_meas) + Ki·∫(v_target − v_meas)dt

| 요소 | 설계 결정 |
|---|---|
| FF (피드포워드) | R2 캘리브레이션의 duty↔정상속도 1차 근사 + 기동 데드밴드 오프셋. **PI는 FF의 잔차만 메우게 되어 튜닝 난이도가 극적으로 내려간다** |
| Anti-windup | 출력 포화(0/100%) 시 적분 동결(clamping). 필수 — 없으면 기동·정지 때마다 오버슈트 |
| 목표 슬루 제한 | 예: 200 cm/s² — 급격한 목표 변화로 인한 포화·바퀴 미끄러짐 방지 |
| D항 | **넣지 않는다.** 속도루프의 D는 측정 잡음만 증폭 — PI로 시작해 부족할 때 검토 |
| dt | 틱 실측값 사용 (§2.3의 하이브리드 루프와 정합) |

**캘리브레이션·튜닝 절차 (순서 고정):**

1. 개루프 duty 스윕 {25, 35, 50, 65, 80}% → 정상 속도 기록 → FF 직선 적합 + 기동 최소 duty(데드밴드) 파악.
2. Kp만 인가: 스텝 목표(30→60 cm/s)에서 진동 직전까지 상향.
3. Ki 추가: 정상편차 소거, 오버슈트 < 15% 유지.
4. **저충 배터리로 재검** — FF 오차를 I항이 흡수하는지 확인. 이것이 통과되면 "배터리 무관 등속"이라는 Phase 1 ②의 존재 이유가 실증된다.

**기존 노브 마이그레이션:** drive.h의 %duty 노브(DRIVE_SPEED 32, TURN_SPEED 42 등)는 R2 캘리브레이션 표를 이용해 cm/s 단위로 재정의한다. 전환기에는 feature flag(SPEED_LOOP 0/1)로 구/신 병행 — 어떤 마일스톤에서든 직전 동작으로 즉시 롤백 가능해야 한다.

## 4. 텔레메트리 통신 프로토콜 설계

### 4.1 설계 원칙 — MIT 앱 인벤터의 현실에 맞춘다

1. **텍스트 + 구분자 프레이밍.** 앱 인벤터 BluetoothClient는 DelimiterByte(=10, LF) 기반 `ReceiveText(-1)`가 가장 안정적이다. 바이너리 프로토콜은 블록 코딩에서 파싱 지옥.
2. **정수만 보낸다.** 소수는 ×10 스케일 정수로 (heading 87.3° → 873, 앱에서 ÷10). 파싱 실패 원인 제거 + 프레임 단축 + 펌웨어 float printf 회피(스택 절약).
3. **고정 필드 수 + 헤더 문자.** v1 무결성 검사는 "필드 수 = 10, 첫 필드 = T" 두 가지로 충분하다. 체크섬(NMEA식 `*XX`)은 v2 옵션.
4. **10 Hz.** 앱 인벤터 Clock·BT 스택의 현실적 처리 상한이며, 그 이상은 사람 눈으로 구분도 안 된다.
5. **TX/RX 문법 분리.** 프레임(T,...)과 명령(#... / 레거시 1바이트)은 문법이 달라 서로 간섭하지 않는다.

### 4.2 TX 프레임 정의 (펌웨어 → 앱)

포맷: `T,<t_ms>,<f>,<L>,<R>,<h>,<vL>,<vR>,<st>,<fl>` + LF(\n)

| # | 필드 | 타입/범위 | 의미 |
|---|---|---|---|
| 1 | `T` | 문자 | 헤더 겸 프로토콜 버전 앵커 (포맷 개정 시 `T2`) |
| 2 | t_ms | uint32 | HAL_GetTick — 앱 그래프 x축, 지연 진단용 |
| 3 | f | 0~120 | 전방 초음파 [cm] (120 = 무효/무한) |
| 4 | L | 0~1000 | 좌측 ToF [mm] (1000 = 캡) |
| 5 | R | 0~1000 | 우측 ToF [mm] |
| 6 | h | 0~3599 | heading ×10 [0.1° 단위] |
| 7 | vL | int | 좌 휠 속도 [cm/s], 음수 = 후진(명령 방향 기준) |
| 8 | vR | int | 우 휠 속도 [cm/s] |
| 9 | st | 0~7 | 주행 상태: 0 CRUISE, 1 BRAKE, 2 SPIN, 3 REVERSE, 4 HOLD, 5 SIDE_AVOID, 6 CORNER, 7 MANUAL — **drive.c enum과 1:1 고정, 순서 변경 금지** |
| 10 | fl | 비트필드 | b0 f_valid, b1 side_valid, b2 imu_live, b3 sys_power, b4 sys_mode |

예시 3종:

```
T,45210,52,148,153,873,41,40,0,15      ← 직선 순항: 전방 52cm, 좌148/우153mm(약간 좌측 치우침),
                                          87.3°, 41/40cm/s, CRUISE, 플래그 0b01111
T,46830,19,102,412,882,28,44,6,15      ← 코너: 전방 19cm 근접, 우측 트임(412mm), CORNER 상태
T,51002,120,1000,1000,1804,0,0,7,24    ← 수동모드 정지: 전 센서 무한/캡, 180.4°, 정지
```

최악 프레임 길이 ≈ 47바이트.

### 4.3 RX 명령 (앱 → 펌웨어) — 레거시 공존 설계

| 종류 | 문법 | 예 | 처리 규칙 |
|---|---|---|---|
| 레거시 (유지) | 1바이트 문자 | `1` 전원 ON, `0` OFF, 모드 문자 | 기존 앱·버튼 그대로 호환 — 즉시 처리 |
| 신규 튜닝 | `#KEY=VAL` + LF | `#VT=45` `#KP=0.60` `#TEL=0` `#HZ=5` | `#` 수신 시 라인모드 진입(버퍼 ≤ 24 B, 500 ms 타임아웃 시 버퍼 리셋), LF에서 파싱 후 원자 반영 |

이것이 R6 "주행 중 재플래시 없는 게인 튜닝"의 기반이며, 튜닝 1회 반복 비용을 수 분(빌드+플래시)에서 수 초로 줄인다.

### 4.4 대역폭 예산

| 보레이트 | 전송률 | 47 B 프레임 소요 | 10 Hz 점유율 | 판정 |
|---|---|---|---|---|
| 9600 (현재) | 960 B/s | 49 ms | **49%** | 동작은 하나 여유가 없다. TX는 반드시 IT 방식(블로킹 금지) |
| **115200 (권장)** | 11,520 B/s | 4.1 ms | 4% | R3에서 승격: HC-06은 `AT+BAUD8`, HC-05는 `AT+UART=115200,0,0` + .ioc 보레이트 동기 |

### 4.5 앱 인벤터 수신 파이프라인 (블록 설계 순서)

1. **연결 전** (Screen.Initialize): `BluetoothClient1.DelimiterByte = 10` 설정.
2. ListPicker로 페어링 목록 표시·연결.
3. `Clock1`(Interval 50 ms) Timer 이벤트: IsConnected일 때 → **while** `BytesAvailableToReceive > 0`: `line ← ReceiveText(-1)` → `parts ← split text at ","` → **if (length of list parts = 10) and (select item 1 = "T")** 일 때만 화면 갱신. heading 라벨 = item 6 ÷ 10.
4. 송신: 버튼/슬라이더 → `SendText( join("#VT=", 슬라이더값, "\n") )`.

흔한 함정 3가지: ① DelimiterByte는 **연결 전에** 설정해야 적용된다. ② numberOfBytes에 **-1**을 줘야 딜리미터 모드다(다른 값은 바이트 수 모드). ③ while로 백로그를 소진하지 않으면 프레임 발행이 Clock보다 빠를 때 표시가 수 초씩 뒤처지는 지연이 누적된다.

## 5. 개발 로드맵

**핵심 결정: 텔레메트리(R3)를 속도 PID(R4)보다 앞에 놓는다.** 관측 수단 없이 PI를 튜닝하면 "굴려 보고 감으로" 반복하게 된다. 텔레메트리가 먼저 있으면 R4의 스텝응답, R5의 지그재그 진폭이 전부 숫자와 그래프로 보인다. **관측 가능성이 튜닝 비용을 지배한다.**

| 단계 | 이름 | 주요 작업 | 완료 기준 (DoD) | 롤백 수단 |
|---|---|---|---|---|
| R0 (0.5일) | 자원 확정 | A안 배선(PA7/PB0), 엔코더 디스크·포토센서 장착, .ioc에 TIM3 CH2/CH3 IC 추가 | 디스크 수동 회전 시 SWD Live Watch로 캡처값 변화 관측 | 배선뿐 — 코드 영향 0 |
| R1 (1일) | 캡처 드라이버 | 에지 기록 + ovf 16→32b 확장 + edge_seq + 0속 타임아웃 | 손 회전 저속→고속에서 주기 단조 변화, 정지 후 100 ms 내 0속 판정 | 모듈 미호출 |
| R2 (1일) | 속도 검증·캘리브 | 주기→cm/s 환산·필터, duty 스윕 {25~80}% → FF 표 | 2 m 직선 스톱워치 실측 vs 텔레메트리 계산 속도 오차 < 5% | — |
| R3 (1일) | 텔레메트리 | TX 프레임 + BT 태스크 재구조 + 115200 승격 + 앱 v1 | 60초간 10 Hz 프레임 필드검증 실패율 < 1%, 명령 왕복 체감 < 200 ms | `#TEL=0` |
| R4 (1~2일) | 속도 PI | SPEED_LOOP=1, PI+FF+anti-windup | 스텝 30→60 cm/s: 정착 < 300 ms, 오버슈트 < 15%, 정상오차 ±3 cm/s. **만충/저충 배터리에서 동일 속도 유지** | SPEED_LOOP=0 |
| R5 (1~2일) | 조향 캐스케이드 | drive.c 출력 Δduty → Δcm/s 전환, 노브 cm/s 재정의, 게인 재튜닝 | 1.6 m 직선 횡진폭 영상 비교 반감, 벽 접촉 0회, 랩타임 30.9 s → 25 s권 | 플래그 전환 |
| R6 (0.5일, 선택) | 원격 튜닝 | `#KEY=VAL` 파서 + 게인·목표속도 반영 | 주행 중 KP/VT 변경이 재플래시 없이 즉시 반영 | — |

의존 관계: **R0 → R1 → R2 → R3 → R4 → R5**, R6은 R3 이후 아무 때나 삽입 가능. 모든 단계는 feature flag로 독립 롤백이 가능해야 하며, IWDG fail-safe는 전 구간에서 무장을 유지한다.

## 6. 리스크 레지스터 & 착수 전 체크리스트

### 6.1 리스크 레지스터

| 리스크 | 영향 | 완화책 |
|---|---|---|
| BNO055 ADR을 0x29로 변경 시 우측 ToF(0x52)와 버스 충돌 | I2C1 전면 오동작 | ADR = 0x28 고정 — 금지 항목으로 명문화 (§1.5) |
| 엔코더 신호선 EMI (모터 전원선 인접) | 유령 에지 → 속도 스파이크 → PI 요동 | 배선 분리·트위스트, ICFilter 상향, 주기 median3 (§3.2) |
| 9600 bps 점유 49% + 블로킹 송신 | 명령 지연·프레임 밀림 | IT 송신 강제 + R3에서 115200 승격 |
| BluetoothTask 스택 부족 (프레임 조립) | HardFault | 384 word 상향 + high-water 재측정, float printf 금지 |
| I2C1(100 kHz) 3슬레이브 혼잡 | 샘플 지연 | 현 폴링 주기 유지 시 여유 존재 — 배선이 짧다면 400 kHz 승격 옵션 |
| FF 표의 배터리 전압 드리프트 | duty 과/부족 | I항 여유로 흡수 (R4 DoD로 실증), 주기적 재캘리브 |
| 16-bit 랩 미처리 | 0.16 m/s 미만에서 속도 오판 | ovf 확장을 R1 DoD에 포함 — 생략 불가 |
| XSHUT 전기 사양 (2.8 V 도메인) | ToF 손상/오동작 | 브레이크아웃 레벨시프터 유무 확인, 미내장 시 open-drain + 풀업 |

### 6.2 착수 전 실측·확인 체크리스트

- [ ] 휠 지름·슬롯 수 실측 (본 문서 가정: Ø65 mm / 20슬롯 — §3.2 환산 상수 치환)
- [ ] 포토센서 모듈 출력 형태 확인 (LM393 디지털인지, 오픈컬렉터면 풀업 필요 여부)
- [ ] ToF 브레이크아웃의 레벨시프터·풀업 내장 여부
- [ ] 블루투스 모듈 종류 확인 (HC-06 vs HC-05 — AT 명령 체계가 다름)
- [ ] BNO055 ADR 스트랩 = 0x28 상태 확인 및 하네스에 표기
- [ ] SWD Live Watch에 dbg 미러 확장 항목(period_us, v_meas 등) 등록 계획
